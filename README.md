# Provenance Guard

Provenance Guard is a Flask backend that classifies submitted creative text as more likely to be human-authored or AI-generated, returns a confidence score and a user-facing transparency label, records every decision in a structured audit log, and lets creators appeal a classification.

## Architecture Overview

A submission follows this path:

1. The client sends a JSON request to POST /submit with a text field and a creator_id.
2. The service runs two independent signals over the text: a semantic signal and a stylometric heuristic signal.
3. The signals are combined into a single confidence score.
4. The score is mapped to one of three transparency labels.
5. The original decision is stored in the submissions table and appended to the audit log.
6. A creator can later file an appeal at POST /appeal, which moves the content status to under_review and appends a new log entry with the appeal reasoning.

## Detection Signals

### 1. Semantic LLM-style signal
This signal asks a Groq model to judge whether the text looks more like human-authored or AI-generated prose. It captures broad semantic coherence, tone, and the overall feel of the writing. It can miss short snippets, highly edited output, or cases where the writing is formal but genuinely human.

### 2. Stylometric heuristic signal
This signal measures structural properties such as punctuation density, sentence-length regularity, lexical diversity, and the presence of casual markers like contractions and first-person phrasing. It captures patterns that are easier to quantify than semantics alone. It can miss very short passages and may over-penalize an informal but human author.

## Confidence Scoring

The two signals are combined with a weighted average:

$$
confidence = 0.6 \times semantic\_score + 0.4 \times stylometric\_score
$$

The score is intentionally conservative. A score of 0.95 pushes the result into the high-confidence AI band, while a score near 0.51 stays in the uncertain band. A false positive is treated as more harmful than a false negative, so the system errs on the side of uncertainty rather than overconfidently labeling a human creator as AI.

The thresholds are:

- $confidence \ge 0.65$: high-confidence AI
- $confidence \le 0.35$: high-confidence human
- otherwise: uncertain

### Validation examples

I tested the scoring logic with several deliberately chosen inputs:

- AI-like prose: "Artificial intelligence represents a transformative paradigm shift in modern society. It is important to note that while the benefits of AI are numerous, it is equally essential to consider the ethical implications."
  - Result: confidence 0.70, label: high-confidence AI
- Human-like casual prose: "ok so i finally tried that new ramen place downtown and honestly? underwhelming. the broth was fine but they put WAY too much sodium in it."
  - Result: confidence 0.29, label: high-confidence human
- Borderline formal passage: "The relationship between monetary policy and asset price inflation has been extensively studied in the literature."
  - Result: confidence 0.53, label: uncertain

## Transparency Labels

The exact text shown to readers is:

| Variant | Text |
| --- | --- |
| High-confidence AI | "High confidence this content appears to have been generated by AI." |
| High-confidence human | "High confidence this content appears to have been created by a human." |
| Uncertain | "Uncertain origin: this content could be AI-generated or human-authored." |

These labels are plain-language and visibly change depending on the confidence range rather than just showing a number.

## Appeals Workflow

A creator can file an appeal with a content_id and creator_reasoning. The submission status changes to under_review and a new audit entry is appended that preserves the original decision and the appeal reasoning.

## Rate Limiting

The submission endpoint is rate-limited to 10 requests per minute and 100 requests per day per client. These limits are intentionally generous for a single creator submitting a handful of works, but strict enough to slow down repeated automated flooding or a simple abuse script. The project uses Flask-Limiter with in-memory storage for local development.

## Audit Log

Every submission and appeal is stored in a structured SQLite-backed audit log with a timestamp, content_id, creator_id, event_type, attribution, confidence, signal scores, status, label, and, when present, appeal_reasoning. The route GET /log returns the latest entries as JSON.

### Sample log entries

  "llm_score": 0.87,

```

```

## Known Limitations

This system would likely misclassify a very short poem or a highly repetitive creative piece because the stylometric signal can overfit to regularity and the semantic signal can over-interpret polished phrasing. The signals are useful for broad triage, but they are not a substitute for human review in disputed cases.

## Spec Reflection

The planning document helped a lot because it forced me to define the thresholds before implementation. The biggest divergence from the spec was the fallback behavior for the semantic signal: when the Groq API is unavailable, the app still returns a meaningful result using a deterministic proxy rather than failing outright. That choice makes the service usable in local development and in environments without a live API key.

## AI Usage

1. I asked an AI coding assistant to generate the initial Flask app skeleton and the first signal stub. It produced a clean route structure and a basic semantic scoring function, which I then revised to make the output more explicit and to add storage for submissions and audit entries.
2. I asked the AI assistant to help flesh out the confidence-scoring and appeal logic. The draft used a simpler weighting scheme than I wanted, so I overrode it with the 0.6/0.4 weighting and the conservative uncertainty thresholds described above.

## Stretch Feature: Analytics Dashboard

A lightweight analytics view is available at GET /analytics. It reports the detection pattern (how many submissions landed in each verdict bucket), the appeal rate, and the average confidence across all submissions.

## Portfolio Walkthrough

A short walkthrough is available by running the service locally and sending sample requests to /submit and /appeal, then inspecting /log. The project is designed so a reviewer can see the full flow from submission to transparency label to appeal without any extra tooling.
